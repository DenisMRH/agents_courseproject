## Публичные материалы курса, релевантные проекту

Использованные источники:

- RAG repo: https://github.com/trashchenkov/industrial-ai-agents-rag-course
- Protocols/MCP repo: https://github.com/trashchenkov/industrial-ai-agents-protocols-course
- PDF домашки `RAG и протоколы взаимодействия ИИ-агентов`: https://github.com/trashchenkov/industrial-ai-agents-protocols-course/blob/main/homework_mcp_knowledge_base.pdf

Из RAG repo:

- ожидаемый стек курса: Python `3.12`/`3.11`, GigaChat, embeddings `EmbeddingsGigaR`, Chroma/BM25/Kuzu, LangChain/Deep Agents, Phoenix tracing;
- тема RAG включает ingestion pipeline, очистку и чанкинг документов, Chroma и BM25, retrieval/generation, hybrid search, reranking, compression, agentic RAG, GraphRAG, файловую working memory.

Из Protocols/MCP repo:

- тема протоколов продолжает RAG: агент сначала работает с внешней базой знаний, затем подключает внешние capability providers через MCP/tools/resources/prompts;
- материалы включают MCP basics, один host + несколько MCP provider-ов, skills pattern, A2A RAG agent;
- для GigaChat используются переменные `GIGACHAT_CREDENTIALS`, `GIGACHAT_SCOPE`, `GIGACHAT_MODEL`, `GIGACHAT_EMBEDDINGS_MODEL`.

Из PDF по RAG+MCP:

- требуется связка: документы -> фрагменты -> метаданные -> индекс -> MCP-сервер/инструмент -> top-k результатов поиска;
- корпус: 5-10 документов или один большой документ с достаточным числом смысловых фрагментов;
- метаданные фрагмента: `document_id`, `chunk_id`, `source`, желательно `title`, `section`, `page`, `created_at`;
- нужно показать загрузку документов, чанкинг, добавление метаданных, загрузку в хранилище, проверку возврата метаданных;
- нужен готовый MCP-сервер или готовый MCP-инструмент, пример MCP-конфига, инструкция запуска, название search tool, пример ручного вызова или лог;
- поиск должен быть доступен агенту как tool, например через LangChain/LangGraph/Deep Agents и `langchain-mcp-adapters`;
- результат поиска: `query`, `k`, текст/отрывок, `document_id`, `chunk_id`, `source`, `score`/`rank`;
- проверка качества: 15-20 запросов, включая точные факты, термины, вопросы по конкретному документу, широкие смысловые вопросы, вопросы без ответа в корпусе, вопросы с несколькими источниками и вопросы, где важны метаданные;
- сдача должна включать `README.md`, MCP-конфиг, ingest script/notebook, agent demo, search demo, eval report, dependencies;
- критерии PDF: 20% корпус/RAG, 20% MCP-сервер, 20% MCP-инструмент в агенте, 15% top-k с метаданными, 15% проверка на 15-20 запросах, 10% воспроизводимая инструкция.

## Что следует считать предметом проекта

Проект не должен быть просто чат-ботом про кредитование. По формулировке варианта и материалам курса он должен демонстрировать промышленную связку:

1. Агентное поведение и промпты: системные инструкции, маршрутизация намерений, контроль уверенности, запрет на неподтвержденные решения.
2. RAG по нормативным документам: поиск по правилам/регламентам кредитования МСБ с цитированием источников.
3. Инструменты: доступ к клиентской БД, например профиль клиента, заявки, активные кредиты, документы, статусы.
4. Эскалация на оператора: создание обращения/тикета с кратким резюме, причиной эскалации, контекстом клиента и найденными источниками.
5. Проверяемость: eval-набор, трассировка, логи tool calls, ручная оценка качества и сценарии отказа.

## Рекомендуемая архитектура

Компоненты:

- UI: Telegram bot, Streamlit/Gradio или CLI demo. Для защиты достаточно управляемого demo, если оно показывает полный flow.
- Agent Orchestrator: LangGraph или LangChain agent с tool calling.
- Prompt layer:
  - system prompt с ролью агента поддержки кредитования МСБ;
  - policy prompt: что можно отвечать, когда нужны источники, когда нельзя давать решение;
  - escalation prompt/classifier;
  - response template с цитатами и next steps.
- RAG service:
  - loader для нормативных документов;
  - chunker со стабильными `document_id`/`chunk_id`;
  - vector index, лучше Chroma/Qdrant;
  - hybrid retrieval/rerank при наличии времени;
  - source-grounded answer generation.
- MCP/tools boundary:
  - `search_normative_docs(query, k)`;
  - `get_client_profile(client_id)`;
  - `get_credit_applications(client_id)`;
  - `get_active_loans(client_id)`;
  - `get_missing_documents(application_id)`;
  - `create_operator_ticket(client_id, reason, summary, evidence)`;
  - опционально `get_product_terms(product_id)`.
- Client DB:
  - `clients`;
  - `credit_applications`;
  - `loans`;
  - `documents`;
  - `support_tickets`;
  - `products`/`tariffs`, если они не только в нормативных документах.
- Observability:
  - trace id на диалог;
  - лог входного вопроса, классификации intent, RAG query, top-k chunks, tool calls, final answer, escalation decision;
  - метрики: retrieval hit@k, answer groundedness, tool correctness, escalation precision/recall.

## Основной диалоговый flow

1. Получить вопрос клиента.
2. Классифицировать intent:
   - общий вопрос по кредитованию;
   - вопрос по статусу заявки;
   - вопрос по условиям/документам;
   - вопрос по платежам/задолженности;
   - жалоба/реструктуризация/спорная ситуация;
   - вне темы.
3. Проверить, нужен ли клиентский контекст.
4. Если нужен, вызвать tools к БД.
5. Выполнить RAG по нормативным документам.
6. Сформировать ответ:
   - кратко;
   - с указанием источников/фрагментов;
   - с ограничением уверенности;
   - без обещаний одобрения кредита, если это не подтверждено данными.
7. Принять решение об эскалации.
8. Если эскалация нужна, создать тикет оператору и отдать клиенту понятный статус.

## Условия эскалации

Эскалировать обязательно:

- клиент прямо просит оператора;
- вопрос про отказ, спор, жалобу, реструктуризацию, просрочку, изменение персональных/юридических данных;
- запрос требует индивидуального кредитного решения или изменения условий договора;
- не хватает данных в БД или данные противоречат нормативным документам;
- RAG не нашел надежного источника или confidence ниже порога;
- пользователь сообщает о мошенничестве, компрометации, ошибочном списании, юридическом споре;
- вопрос вне зоны агента, но связан с клиентским обслуживанием.

Формат тикета оператору:

- `client_id`;
- имя/сегмент клиента, если доступно;
- тема обращения;
- краткое резюме диалога;
- причина эскалации;
- уже проверенные данные БД;
- найденные нормативные источники;
- последние сообщения клиента;
- желаемое действие оператора.

## Сценарии для демонстрации на защите

Минимальный набор demo:

1. Общий вопрос: "Какие документы нужны ИП для заявки на кредит?" Агент отвечает из RAG с источниками.
2. Клиентский статус: "Что с моей заявкой?" Агент вызывает БД, находит заявку, сопоставляет с правилами, показывает недостающие документы.
3. Смешанный вопрос: "Почему запросили поручителя?" Агент использует нормативный поиск и данные клиента/заявки.
4. Эскалация: "Мне отказали, я хочу пересмотр/жалобу" или "Нужно изменить условия по действующему кредиту". Агент не решает сам, создает тикет оператору.
5. Нет ответа в базе знаний: агент честно сообщает, что не нашел подтверждения, и эскалирует или предлагает уточнить.

## Eval-набор

Для защиты стоит подготовить 15-20 запросов:

- 5 общих нормативных вопросов;
- 5 клиентских вопросов с tool calls;
- 3-4 вопроса на эскалацию;
- 2-3 out-of-scope/no-answer;
- 2 вопроса, где нужны несколько источников.

Поля eval table:

- `id`;
- `query`;
- `expected_intent`;
- `expected_tools`;
- `expected_sources`;
- `expected_escalation`;
- `actual_tools`;
- `top_k_chunks`;
- `answer_ok`;
- `groundedness`;
- `comment`.
