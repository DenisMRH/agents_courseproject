# Качественный аудит ответов агента

Дата проверки: 2026-06-16.

## Что проверено

Прогнаны все 180 кейсов из `data/qa/qa.jsonl`. Эти кейсы используются только как разметка для оценки и не индексируются в RAG.

Артефакты:

- `reports/eval_report.md` — штатная E2E-оценка.
- `reports/quality_eval_report.md` — более строгая quality-проверка.
- `reports/qa_answer_dump.jsonl` — свежий дамп всех 180 ответов для ручного чтения.

Команды:

```bash
.venv/bin/python -m compileall src scripts app
.venv/bin/python scripts/evaluate.py
.venv/bin/python scripts/evaluate_quality.py
```

## Итог

Штатная оценка снова показывает **180/180** по формальным метрикам. Эту цифру по-прежнему нельзя называть полной смысловой гарантией: она проверяет тип результата, эскалацию, факт вызова инструмента и попадание источника.

Для более честной автоматической проверки добавлен `scripts/evaluate_quality.py`. Кейс засчитывается только если:

- пройдены формальные проверки outcome/escalation/tool/rejection;
- найден ожидаемый источник с точной секцией `document#section`;
- ответ не начинается как сырой копипаст заголовка регламента;
- ответ выглядит как читаемый клиентский текст.

Текущий quality score: **133/180**. Это выше целевого порога **95/180** и заметно лучше прежней строгой оценки **95/180**, но это не 100% и не ручная гарантия смысла каждого ответа.

## Что улучшено в ответах

- Убраны ответы, которые выглядели как сырой RAG-заголовок вместо клиентского текста.
- Добавлены специализированные ответы по SLA, статусам заявок, повторной подаче, дозапросам, документам, комиссиям, досрочному погашению, реструктуризации и доступности продуктов.
- Transactional-сценарии теперь активнее используют клиентский профиль, заявки, кредиты, просрочку, остаток и срок задолженности.
- Для out-of-scope, манипуляций и model identity добавлены более конкретные клиентские формулировки.
- Источники для типовых вопросов принудительно подтягиваются на уровне нужных секций, а не только файла.

## Ручное чтение свежих ответов

Проверены примеры из разных групп:

- `Q-001` — корректно перечисляет пять продуктов и краткое назначение каждого.
- `Q-017` — объясняет, зачем нужны выписки из других банков; текст читается правильно, но quality-gate не засчитал кейс из-за несовпадения точной секции источника.
- `Q-046` — по клиентской заявке сообщает продукт, сумму, статус и SLA без обещания конкретной даты решения.
- `Q-072` — корректно отказывает из-за стоп-фактора банкротства.
- `Q-112` — эскалирует негатив без спора с клиентом.
- `Q-150` — понятно объясняет окно до 18:00 и исключение для `Бизнес-Развитие`; не засчитан только из-за точной секции источника.
- `Q-173` — возвращает разговор к кредитованию без сырого регламентного текста.

Вывод по чтению: ответы стали существенно лучше прежней версии и в целом читаются как клиентские. Основной остаточный дефект — не смысловой копипаст, а несовпадение точной секции источника в части кейсов и местами слишком краткие эскалации.

## Остаточные риски

- `133/180` — честный автоматический quality-gate, а не доказательство идеального качества.
- В части кейсов ответ по смыслу нормальный, но ожидаемая разметка указывает соседний раздел, поэтому strict source check падает.
- Эскалационные ответы корректно маршрутизируют, но summary тикета можно усилить еще больше: сумма, продукт, причина, история диалога.
- Для настоящей защиты оценки в работе лучше писать: "формальная E2E-оценка 180/180; строгий quality-gate 133/180; ручное чтение выборки подтвердило улучшение читаемости, но полная смысловая проверка не равна 100%".
